## 一、结论速览（TL;DR）

1. **求助必须是"带状态的工单"，不是"一次性消息"。** 全球最成熟的协调平台（Crisis Cleanup、Sahana Eden、卓明）都把每条求助建模成有唯一 ID、有状态机（待认领/已认领/救援中/已获救）、可回写、可去重的 case。我方求助墙目前更接近"消息流"，这是 P0 必须改造的地基。
2. **黄金救援期靠"低门槛入口 + 专业中台分工"。** 众包广撒网采集（文档/超话/表单/SMS/语音），专业组织电话核实 + 分级 + 派单。郑州"救命文档"和卓明的组合是中文语境下最直接的范式。
3. **定位即生命线。** 文本求助自动转经纬度 + 地图化，是"找不到人"这一头号痛点的解药。我方地图能力层（模块 B）应把这一点当作硬需求而非阶段 2。
4. **脆弱人群要"双轨数据 + 被动/代报"。** 罕见病/独居老人平时预登记（加密档案），灾时增量补齐；很多老弱病残孕根本无法自主拍照求助，必须支持第三方代报与被动触发。这正是我方"罕见病 + 方言弱势"差异化定位的护城河。
5. **四个坑会直接害死平台：信息真伪失守、隐私裸奔、重复求助、信息过期不维护。** 水滴筹是审核失守透支信任的反面镜鉴；郑州文档是隐私裸奔的教训。这些必须在架构层用机制（而非人力自觉）解决。

## 三、"帮助灾民"最有效的 7 个通用设计模式

> 从 45 个平台/案例横向提炼。这 7 条是反复出现、被多国多场灾害验证的"最大公约数"。

**模式 1 · 求助 = 带状态的工单，不是消息（去重是第一要务）**
把每条求助建成有唯一 ID、有状态机（待认领→已认领→救援中→已获救/已关闭）、可回写、可去重合并的 case。Crisis Cleanup 的"认领+锁定"、Sahana 的统一 message log、卓明/腾讯小程序的"获救即闭环"都指向同一件事：**灾害协调最大的痛点是漏救与重复救援**（同一人被多方转发、接好几拨电话、救援队电话永远占线），而工单化 + 去重是唯一解药。

**模式 2 · 低门槛多通道入口 + 专业中台分工核实分级**
众包广撒网采集（文档/超话/表单/SMS/语音/图标），专业角色（调度员/接线员）电话核实 + 按 1-4 级紧急度分级 + 派单。郑州"救命文档"×卓明、CrowdSource Rescue 都证明：**采集要极低门槛、快；核实分级要专业、准**。二者分工才既快又准。入口要覆盖不识字/方言/老年/无智能机/断电断网人群——SMS、语音、图标化是刚需，不是加分项。

**模式 3 · 定位即生命线：文本/语音/照片自动转经纬度 + 地图化**
"找不到人"是仅次于漏救的第二痛点。腾讯地图×卓明把文本求助自动转经纬度生成互助地图；Cajun Navy 的"表单→地图"极简闭环；Zello 语音附带定位。台风内涝场景还要叠加**众包积水/路况地图**（京津冀骑手上传深度→导航提示绕行）。每条求助落库时应强制带时间戳 + 坐标。

**模式 4 · 脆弱度 × 紧急度 双维度优先级排序 + 高危标签**
所有成功案例都做优先级：孕妇/老人/婴幼儿/急缺药/被困久 = 高危先救。进阶做法（GiveDirectly、UNDRR WG-SS）是**脆弱度（罕见病/独居/功能困难度）× 紧急度**双维度打分作为派单排序核心，并用国际标准的"6 维功能困难度"（视/听/行动/认知/自理/沟通）去污名化打标——听障走沟通/手语通道、行动困难走搬运优先、罕见病走专病基金会路由。

**模式 5 · 双轨数据模型：平时预登记（加密档案）+ 灾时增量采集**
学 Florida SNR + RapidSOS：罕见病/慢病/独居老人平时建加密健康档案（病种、依赖设备、冷链药、禁忌、对接基金会、方言/听障标签），灾时 AI 只需用一张照片增量补齐位置/现场状态即秒出需求单，避免最脆弱的人临场重填。隐私按"平时加密、灾时按授权解锁"分级。同时支持**第三方代报 + 被动传感触发**（日本 AAL）——很多老弱病残孕根本无法自主发起求助。

**模式 6 · 全程无障碍多模态双向闭环（采集与回执都要无障碍）**
不只采集端无障碍（图标分步/方言语音/代报/被动触发），**回执/指令端也必须多模态**（救援进度、疏散指令用手语短视频、可视图标、振动回传给听障用户——Alertus）。方言人群要能用母语发声并收到母语回执（TWB/Gamayun 双向沟通）。需求单加"首选接收模态"字段。最脆弱人群保留人工坐席兜底，AI 只增效不做唯一入口。

**模式 7 · 权威分层去噪 + 主动触达 + 常态化组织网络**
（a）权威可信信息区（官方审核后推权威通知：撤离/庇护所开放/医疗点）与用户众包上报区**分层**，防谣言（ReliefWeb 策展 × Ushahidi AI 分诊）。（b）**主动触达**而非被动等待：结合台风影响范围 + 用户位置主动推报平安/求助入口（Meta Safety Check），未装 App 者用小区广播秒级覆盖（CAP+Cell Broadcast）。（c）**平时就建**本地救援队/物资点/避难所台账，灾时一键激活 + 提前备灾（壹基金），远胜临时拉群。

## 四、本平台应优先吸收的功能与设计建议（P0/P1/P2）

> 结合我方**已有模块**（预报/电话簿/求助互助/灾后指南）与**在建模块**（AI 拍照求助 `sos`/地图 `map`/物资站点 `stations`/罕见病 `rare`），落到具体交付。P0=灾时不做会害人/漏救；P1=显著提升有效性；P2=增强与生态。

### P0 · 地基级（不做会直接害人或漏救）

**P0-1 求助墙工单化 + 唯一 ID + 状态机 + 去重合并**
- 把 `help_requests` 从"消息流"升级为工单：唯一 `id`、状态机 `pending→claimed→rescuing→rescued/closed`、`claimed_by`（认领的救援队/志愿者）、`updated_at`。
- 求助卡片显示三态（待认领/已认领/已获救），列表默认按"脆弱度×紧急度"降序。
- **认领+锁定去重**（抄 Crisis Cleanup）：一条 case 被认领后对其他救援方置灰"已有人处理"，支持人工合并疑似重复（同手机号/同定位/同描述）。
- 求助者与志愿者**双向可回写状态**，"已获救"一键闭环——这是抑制重复救援的核心（郑州文档/京津冀小程序共识）。
- 关联：这是模块 A(`sos`) 产出的 `HelpRecord` 落地的前提，须先于 AI 拍照上线。

**P0-2 定位强制化：每条求助带时间戳 + GCJ-02 坐标 + 一键导航**
- `sos` 与求助墙落库强制 `lat/lng`（无坐标不可提交或标"定位缺失"高危待补）；文本/语音/照片求助尽力自动转经纬度。
- 地图能力层（模块 B）的"高德/腾讯导航双按钮"从阶段 2 **提前到 P0**——"找不到人"是头号痛点，导航前往不是锦上添花。
- 台风内涝叠加**众包积水/路况图层**（复用骑手/志愿者上传的积水深度）提示绕行（抄京津冀）。

**P0-3 信息真伪与核实纪律落到工单（延续我方现有 `VerifyLevel`）**
- 我方已有 `VerifyLevel = verified|structural|media|unverified` 与 `VerifyBadge`，把它**贯穿到求助工单**：每条求助显式标核实态，未核实不进"已派单"高优先队列。
- 引入"调度员/接线员"角色（卓明双部门制）：多源交叉核实（电话+定位+现场照片+熟人佐证），核实状态与来源可追溯。
- 水滴筹镜鉴：绝不把真伪责任甩给求助者个人；宁可标"待核实"也不杜撰（与我方电话簿纪律一致）。

**P0-4 隐私分级 + 汛后自动脱敏销毁（郑州文档教训）**
- 字段最小化；姓名/手机号/微信/住址/生理特征等**对救援方按需授权可见**，公开地图只显示脱敏点位与紧急度。
- 罕见病档案"平时加密、灾时按授权解锁"（抄 Florida SNR）。
- **汛情结束自动抹除个人信息**（保留脱敏救援复盘数据），并在提交页明示留存与销毁政策 + "登记≠保证救援"话术。

### P1 · 显著提升有效性

**P1-1 AI 拍照/一句话求助：融合"图标分步 + 方言语音 + 病种路由"**（模块 A/E）
- 文本分支抄大埔的"AI 解析群聊乱文本→结构化"；补**图标分步问答**（accesSOS）作为拍照失败/无网降级：先点图标（医疗/被困/缺药/停电），AI 再融合照片+图标+一句话出结构化求助单。
- 方言语音口述→转写→结构化（TWB/Gamayun 思路）；AI 对缺失字段主动追问补全。
- 需求单含 `禁用/慎用` 急救禁忌字段（MedicAlert）；病种字段作**路由键**自动匹配对应罕见病基金会/专病组织，触发特殊药/冷链/营养配方/远程专家（对应模块 E 已规划的"抄送罕见病社区"）。

**P1-2 脆弱人群双轨数据 + 第三方代报 + 被动触发**（模块 E）
- 平时预登记加密健康档案（病种/依赖设备/冷链药/禁忌/主诊/对接基金会/方言听障标签），字段对齐**华盛顿组 6 维功能困难度**（去污名化、国际可比）。
- 求助单支持来源类型 `自报/代报/被动`——支持家属/网格员**代拍代报**独居老人/认知障碍者（补上"需本人拍照"的最大盲区）。
- 派单排序 = **脆弱度 × 紧急度** 双维度打分。

**P1-3 邻里互助层 + 求/供同图双态图钉**（扩展现有 `/aid`）
- 现有求助互助已有 `help/volunteer/donate`，增设"邻里互助"：受灾者(红)与有余力居民(绿)同一地图标点，就近撮合（送水/共享发电/临时安置），社区尺度闭环减轻专业队压力（Nextdoor/Mutual Aid Hub）。
- 结合我方物资站点（模块 D）把避难所/物资点/医疗点作固定叠加图层。

**P1-4 台风预报叠加脆弱人群图层 + 主动触达**（模块预报 + `rare`）
- 对接 GDACS 式台风分级预警思路（我方预报页已直达官方数据源），把罕见病/独居老人/方言人群分布叠到台风影响范围，**提前锁定高危做点对点通知**。
- 结合影响范围+用户位置**主动推**"一键报平安/求助"入口（Meta Safety Check），而非等灾民来找。

**P1-5 离线兜底 + 语音一键 SOS + 低带宽极简页**（全站）
- 台风常停电断网，`sos` 与入户排查支持**离线采集、联网同步**（KoBoToolbox）；提供极简低带宽页面。
- "语音一键求助 = 录音+自动定位+提优先级"（Zello 紧急按钮）——方言/不识字/极端应激下比打字救命。

**P1-6 灾后指南扩"离线急救/罕见病应急处置卡" + 动态待办清单**（扩展 `/guide`）
- 现有灾后指南补**离线可用的应急处置卡**（癫痫发作/低血糖/缺药替代/涉水安全），供家属/邻里照护罕见病人（Red Cross）。
- 把复杂救援/返家步骤做成**动态待办清单**"现在该做什么"，降低灾民认知负担。

### P2 · 增强与生态

**P2-1 寻人/报平安模块**（离散家庭高频刚需，Google Person Finder），用可互通数据标准，含方言语音/图标化界面让老人也能报平安或被家人查询。
**P2-2 内嵌交互多点地图 + 救援队实时位置图层**（模块 B 阶段 2 / 模块 C），前方专职回传现场缺口的反向通道（蓝天救援），无 key 时降级为列表+导航按钮。
**P2-3 数据互通与机构侧复用**：用类 HXL 字段标准让民政/卫健/气象/残联/志愿者数据对齐；结构化需求单按可复用字段输出供 NGO/基金会/政府汇聚（DEEP）；每次台风事件策展专属情境数据页。
**P2-4 常态化组织网络 + 提前备灾**（壹基金）：平时登记本地救援队/物资点/避难所台账，灾时一键激活。
**P2-5 微信生态适配**：导出"图片转发"（大埔）适配微信群传播；把社媒（微博超话/微信群）当引流入口，求助进系统后统一去重核实。
**P2-6 地点别名映射**（大埔）：广西村屯俗名/方言地名→标准地名+坐标映射表，解决山区"报了地名找不到"。

## 五、必须避免的坑（附机制化对策）

> 这些坑在几乎每场灾害都重复出现。关键认知：**必须用架构机制解决，不能靠人力自觉。**

| 坑 | 典型教训 | 后果 | 机制化对策（落到我方） |
|---|---|---|---|
| **① 信息真伪失守** | 水滴筹审核形同虚设、虚假证明即过审；未核实求助引发误调度 | 虚假/夸大挤占黄金救援资源，"狼来了"透支公众信任，平台失信崩塌 | 多源交叉核实(电话+定位+现场照片+熟人佐证)；`VerifyLevel` 贯穿工单，未核实不进高优先派单队列；来源可追溯；无法确认宁标"待核实"不杜撰(延续我方电话簿纪律)；明确平台责任主体，不甩锅求助者 |
| **② 隐私裸奔** | 郑州文档暴露姓名/手机/微信/工作单位/生理特征；灾后海量个人数据无处置 | 个人信息泄露、二次伤害、法律风险 | 字段最小化；敏感字段对救援方按需授权可见，公开地图只脱敏点位；罕见病档案"平时加密、灾时授权解锁"；**汛后自动抹除个人信息**(留脱敏复盘)；提交页明示留存/销毁政策 |
| **③ 重复求助** | 同一人被多方转发、接好几拨电话、救援队电话永远占线；机器爬取多为失效信息 | 救援力量重复扑向同一户，真正的漏救者无人问津；资源空转 | 工单唯一 ID + 去重合并(同手机/同定位/同描述)；**认领+锁定**(Crisis Cleanup)一户被认领即对他人置灰；**获救即闭环**双向状态回写；地理围栏限灾区本地上传(卓明)砍掉外地搬运 |
| **④ 信息过期不维护** | 机器爬取的求助许多早已失效；安置点信息靠零散网络传播、无官方实时源 | 救援队按过期信息空跑；灾民按错误安置点信息奔波 | 每条求助时间戳 + **自动过期机制** + 求助者/志愿者双向状态更新；安置点/物资点做**官方权威实时源**(补京津冀暴露的空白)，标注更新时间与数据来源 |
| **⑤ 协作不畅/组织散落** | 多个志愿组织互不通气、重复劳动；社会力量散落 | 重复救援 + 覆盖盲区并存 | 统一中台 + 开放数据共享，跨组织可见同一份已核实名单；省级枢纽编织本地组织(壹基金)；前后方专职对接回传缺口(蓝天) |
| **⑥ 透明度不足加剧焦虑** | 求助者不知"我的求助走到哪、救援队离我多远" | 焦虑→重复提交→加剧①③ | 给求助者**可见进度反馈**(已核实/已派单/救援中/已获救)；这本身也是抑制重复提交的手段 |
| **⑦ 只做富前端/依赖装 App** | 台风停电断网普遍；假设人人有智能机、会打字、装了 App | 最脆弱人群(老年/方言/无智能机/断网)被系统性漏掉 | 离线采集联网同步(KoBo)；SMS/语音/图标兜底；小区广播秒级覆盖未装 App 者(CAP)；极简低带宽页 |
| **⑧ 商业化侵蚀公益/KPI 压垮审核** | 水滴筹资金池不透明、地推冲量腐蚀审核质量 | 公益属性质疑、审核崩坏 | 明确非营利定位与责任主体；审核质量优先于冲量；资金/物资流向透明(我方捐赠模块已有防骗提示,延续) |

## 六、一句话行动纲领

> **把求助从"消息"升级为"带状态、可去重、带坐标、按脆弱度×紧急度排序的工单"，用"低门槛多通道入口 + 专业中台核实分级"喂满它，为罕见病/方言弱势人群做"平时预登记 + 灾时增量 + 代报/被动触发 + 全程无障碍双向闭环"，并用架构机制(而非人力自觉)守住真伪、隐私、去重、时效四条底线。**

这正是我方相对香港大埔原型（紧急临时开发、无审核无权限无隐私）和纯社媒/文档众包（强扩散弱闭环）的差异化护城河：**一个工单化、可核实、隐私可控、且对最脆弱人群友好的台风救援协调系统。**
