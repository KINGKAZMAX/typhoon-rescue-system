# 07 · AI 拍照求助 —— 多模态需求提取功能设计规格

> 关联 PRD：受灾群众（含不便打字、方言沟通困难者）通过【拍照 + 少量文字/语音】提交求助，系统用多模态 AI 自动识别、分类、生成【结构化需求项】并分派。
> 技术栈：Vite + React + TypeScript + Supabase（见 `package.json` / `.env.example`），移动端优先，中国大陆免翻墙可用，前端为静态站不得暴露密钥。
> 本文档产出：`needFields` / `rareDiseaseFields` / `urgencyLevels` / `promptTemplate` / `deployment` / `fallbackUX` 六项设计规格。
> 更新日期：2026-07-11

---

## 0. 设计原则（先于字段设计）

1. **不臆造事实（No-Hallucination First）**：AI 只从「照片 + 文字/语音转写」中能确证的信息填字段；无法确定的一律留空（`null`）并在 `_uncertain` 数组标注，绝不编造门牌号、人数、病名。研究表明：不允许 null 会「逼迫模型幻觉」，因此每个可选字段都必须允许空值并显式提示模型「不确定就填 null / unknown」。
2. **结构化 + 原文并存**：结构化字段用于分派与统计，同时保留 `rawText`（用户原话）与照片 URL，供人工复核，杜绝信息在归纳中丢失。
3. **保守升级紧急度**：涉及生命、被困、断药、婴幼儿/老人时，宁可高估 `urgency`，由人工下调，不由 AI 擅自下调（灾害分诊「动态、就高」原则）。
4. **字段少而扁平**：单条求助控制在 ~20 个核心字段、嵌套 ≤2 层。研究指出 50+ 字段、4+ 层嵌套会显著降低结构化输出质量。
5. **人机协同**：AI 输出是「草稿」，前端需让求助者或救援站一键确认/修正；`confidence` 低于阈值的字段前端高亮待人工确认。
6. **降级永远可用**：AI 不可用（限流、断网、密钥问题）时，引导式表单覆盖同样字段，产出同一套 JSON，保证系统 100% 可提交。

## 1. needFields —— 通用结构化求助字段

一条求助被规整为以下字段。`type` 取值约定：`enum`（单选枚举）、`enum[]`（多选枚举）、`int`、`string`、`bool|null`（三态，允许"不确定"）、`object`。所有字段除 `rawText` 外均允许 `null`。

| key | label | type | desc |
|---|---|---|---|
| `urgency` | 紧急程度 | `enum` | 分级见 §3。取值 `critical`/`high`/`medium`/`low`。涉及生命/被困/断药/婴幼儿默认就高。 |
| `needTypes` | 需求类型 | `enum[]` | 多选：`trapped`(被困) / `medical`(医疗) / `supplies`(物资) / `medication`(断药) / `evacuation`(转移安置) / `missing`(失联寻人) / `rescue_water`(涉水/洪水被困) / `shelter`(临时住所) / `psychological`(情绪支援) / `other`。一条求助可含多项。 |
| `locationText` | 所在位置文本 | `string` | 原文位置描述，如"金秀县桐木镇XX村3组老槐树旁蓝顶房"。不做地理编码，保留原样；模型不得补全未提及的行政区。 |
| `locationDetail` | 位置补充线索 | `string` | 从照片/文字提取的辅助定位线索（门牌、店招、地标、楼层）。无则 null。 |
| `peopleCount` | 现场总人数 | `int` | 需帮助的人数。文字/照片未明确则 null，不猜。 |
| `hasElderly` | 含老人 | `bool\|null` | 是否有 60+ 老人。未提及=null。 |
| `hasChildren` | 含婴幼儿/儿童 | `bool\|null` | 是否有 14 岁以下儿童（尤其 <3 岁婴幼儿，见 `vulnerableNote`）。 |
| `hasPregnant` | 含孕产妇 | `bool\|null` | 是否有孕妇/产妇/月子期。 |
| `hasInjuredSick` | 含伤病员 | `bool\|null` | 是否有受伤或急病者。为 true 时应补 `injuryNote`。 |
| `injuryNote` | 伤病简述 | `string` | 伤情/病情原话或从照片可见的伤情（出血、骨折、昏迷等）。不臆断诊断，只描述可见/可闻事实。 |
| `vulnerableNote` | 脆弱人群补充 | `string` | 残障、失能、精神障碍、独居等需特别关照信息。 |
| `suppliesNeeded` | 所需物资清单 | `object[]` | 每项 `{item, quantity, unit, note}`，如 `{item:"饮用水", quantity:12, unit:"人份/天", note:"含1名婴儿需奶粉"}`。数量不明填 null。 |
| `canMove` | 可否自行行动 | `enum` | `yes`(可自行撤离) / `partial`(需搀扶/部分行动能力) / `no`(完全无法移动/被困) / `unknown`。影响转移方案。 |
| `waterLevel` | 涉水/水位情况 | `enum` | `none` / `ankle`(没脚踝) / `knee`(没膝) / `waist`(没腰) / `chest_above`(齐胸以上/危急) / `unknown`。台风洪涝专用。 |
| `powerComm` | 断电/断网 | `enum[]` | 多选：`power_out`(停电) / `no_signal`(无信号) / `phone_low_battery`(手机没电) / `water_cut`(停水) / `normal` / `unknown`。影响回访方式。 |
| `isRareDisease` | 涉罕见病/慢病 | `bool\|null` | 是否涉及罕见病或需长期用药慢病患者。为 true 时触发 §2 附加字段。 |
| `contact` | 联系方式 | `object` | `{name, phone, altContact, canCallBack}`。电话缺失=null，此时提示前端强制补录。 |
| `confidence` | 整体置信度 | `object` | `{overall: 0~1, low_fields: [字段名...]}`。overall<0.6 前端标"需人工确认"。 |
| `_uncertain` | 不确定项标注 | `string[]` | 模型判断为"不确定/靠推测"的字段名或说明，如 `["peopleCount 系照片粗估","locationText 方言音译不确定"]`。 |
| `rawText` | 用户原话/转写 | `string` | 必填。用户输入文字或语音转写全文，方言不改写，原样保留。 |
| `mediaRefs` | 关联媒体 | `string[]` | 照片/语音在 Supabase Storage 的引用 key，供人工复核。 |

> 分派逻辑（供后续开发参考，不属本文档产出但便于理解字段用途）：`needTypes` → 分派到对应救援队列；`urgency` → 排序；`isRareDisease=true` → 同时抄送罕见病基金会/病友社区（呼应主 PRD "散发向各罕见病患者社区"）。

## 2. rareDiseaseFields —— 罕见病 / 慢病附加字段

当 `isRareDisease=true` 时附加。为对接罕见病基金会数据与药物调配（呼应主 PRD "罕见病患者优先确认患者体征状态、病情确认、药物储备"）而设。**病名/诊断类字段极易幻觉，规则：只登记用户明确说出或药盒/病历照片上可读到的文字，读不到就 null，绝不据体征反推病名。**

| key | label | desc |
|---|---|---|
| `diseaseName` | 病种/诊断名称 | 用户自述或病历/药盒照片可读到的病名（如"庞贝病""血友病A""苯丙酮尿症"）。只登记明确文字，不由症状反推。未知=null。 |
| `diseaseCategory` | 病种大类 | 便于分派社区：`rare_metabolic`(罕见代谢病) / `blood_coagulation`(血液/凝血) / `neuromuscular`(神经肌肉) / `immune` / `renal_dialysis`(肾病透析) / `respiratory_oxygen`(呼吸/需氧) / `endocrine`(如糖尿病/甲状腺) / `other` / `unknown`。 |
| `currentVitals` | 当前体征/状态 | 可观察/自述的当前状态：意识、能否进食、有无抽搐/出血/呼吸困难、体温、血糖值等。只记事实不下诊断。 |
| `medicationsNeeded` | 所需药物及剩余量 | 数组，每项 `{drugName, dose, remainingAmount, runOutDate, source}`。剩余量与"还能撑几天/几针"尤为关键（断药=危急）。药名以照片/自述为准，读不清标注于 `source`。 |
| `needsDialysis` | 是否需透析 | `bool\|null` + `dialysisNote`（末次透析时间、频率、原透析点是否停摆）。透析中断为生命威胁级。 |
| `needsOxygen` | 是否需供氧 | `bool\|null` + `oxygenNote`（制氧机是否依赖电力/停电影响、氧气瓶剩余、流量 L/min）。停电致制氧机停机为危急。 |
| `coldChainMeds` | 冷链药物 | `bool\|null` + `coldChainNote`（需冷藏药物名称、当前是否已断冷/停电、可维持时长）。停电致冷链中断需优先调配冰袋/转移，凝血因子、胰岛素、部分酶替代药为典型。 |
| `deviceDependent` | 依赖医疗设备 | 依赖的设备（呼吸机、胃管、造口、鼻饲泵、监护仪等）及供电/耗材依赖，`{device, powerDependent, consumableNote}`。 |
| `medicalDocs` | 病历/证明资料 | 是否有近期病历/用药清单/诊断证明照片可供基金会核对，`{hasDocs: bool, docRefs: []}`。对接基金会核验用。 |
| `foundationLinked` | 已联系的基金会/病友组织 | 用户已联系或所属的罕见病组织名称（如"病痛挑战基金会""血友之家"）。便于精准散发，未知=null。 |
| `allergyContra` | 过敏/禁忌 | 已知药物过敏或禁忌，避免误配药物。 |

## 3. urgencyLevels —— 紧急程度分级定义

采用与灾害分诊「红/黄/绿」就高原则一致的四级制，映射为红/橙/黄/绿，附「首响时限」与「触发条件」，便于前端着色与队列排序。**AI 只上调不下调**：命中任一"就高触发"即取更高级。

| level | 标签(色) | 首响目标 | 定义与触发条件（命中其一即归入） |
|---|---|---|---|
| `critical` | 危急 (红) | 立即 / <30 分钟 | 生命垂危或分秒级威胁：被困洪水且水位齐胸以上、无法呼吸、大出血/昏迷、透析/供氧/冷链药中断且危及生命、临产、多名婴幼儿被困、结构坍塌困人。 |
| `high` | 紧急 (橙) | <2 小时 | 有明确生命/健康风险但暂稳：被困待转移（水位膝~腰）、急病伤员需就医、断药 24 小时内耗尽、老人/婴幼儿/孕产妇处于恶劣环境、慢病体征恶化。 |
| `medium` | 一般 (黄) | 当日内 | 需求明确但暂无即时生命危险：基本物资短缺（食物/饮用水/药品数日内耗尽）、需转移至安置点但环境安全、失联寻人（无伤情线索）。 |
| `low` | 普通 (绿) | 24–48 小时 / 排期 | 非紧迫需求：生活物资补充、信息咨询、情绪支援、灾后重建协助、非急需物品。 |

> 前端与分派：`critical`/`high` 置顶闪烁并推送值班；罕见病相关的透析/供氧/冷链/断药命中 → 即使体征暂稳也建议 `high` 起步（时间敏感）。分级仅为分派参考，最终由救援调度人工确认。

## 4. promptTemplate —— 多模态大模型中文提示词模板

> 说明：输入 = 用户照片（1~N 张）+ 用户文字或语音转写。输出 = 严格 JSON（§1 + §2 字段）。已内置「方言/模糊描述稳健归纳」「不臆造」「不确定标注」「就高定级」约束。部署时把本模板作为 system/user 提示注入，图片按各厂商多模态格式随附。

```text
你是一个台风洪涝灾害求助信息结构化助手，服务对象是广西受灾群众，其中很多人不便打字、说方言、描述零散。
你的唯一任务：把【用户上传的照片】和【用户的文字/语音转写】整理成一条严格符合下方 JSON Schema 的结构化求助记录，用于救援分派。

【最高原则 —— 必须遵守】
1. 只写你能从照片或文字中确证的信息。看不到、听不清、没说到的，一律填 null（数组填 []），绝不编造。
2. 严禁臆造：不要补全没提到的门牌号/村名/行政区；不要根据症状反推病名；不要猜测人数——不确定就 null。
3. 方言与模糊描述要稳健归纳：把口语、方言词、语气词归纳为标准表述，但不得改变或夸大事实（例："困得凶咧，水都齐腰噜" → waterLevel="waist"，urgency 至少 high）。方言地名按读音音译并在 _uncertain 标注"音译不确定"。
4. 定级就高不就低：凡涉及生命、被困、断药、透析/供氧/冷链中断、婴幼儿/老人/孕产妇、大出血/昏迷/呼吸困难，urgency 从高判定（可上调不可下调）。拿不准时选更紧急的一级。
5. 每一个你靠推测或不确定的字段，把字段名和原因写进 _uncertain 数组；并在 confidence.low_fields 列出低置信字段，给出 confidence.overall（0~1）。
6. rawText 必须原样保留用户原话/转写（含方言），不要改写。
7. 只输出 JSON，不要任何解释、前后缀、markdown 代码块标记。

【枚举取值约定】
- urgency: 见规则4，取 "critical" | "high" | "medium" | "low"
- needTypes(可多选): "trapped"被困 | "medical"医疗 | "supplies"物资 | "medication"断药 | "evacuation"转移安置 | "missing"失联寻人 | "rescue_water"涉水被困 | "shelter"临时住所 | "psychological"情绪支援 | "other"
- canMove: "yes" | "partial" | "no" | "unknown"
- waterLevel: "none" | "ankle"没脚踝 | "knee"没膝 | "waist"没腰 | "chest_above"齐胸以上 | "unknown"
- powerComm(可多选): "power_out"停电 | "no_signal"无信号 | "phone_low_battery"手机没电 | "water_cut"停水 | "normal" | "unknown"
- 三态布尔字段(hasElderly/hasChildren/hasPregnant/hasInjuredSick/isRareDisease/needsDialysis/needsOxygen/coldChainMeds): true | false | null(未提及)

【输出 JSON 结构】（未涉及罕见病时 rareDisease 字段填 null）
{
  "urgency": "high",
  "needTypes": ["trapped","medical"],
  "locationText": "",
  "locationDetail": null,
  "peopleCount": null,
  "hasElderly": null, "hasChildren": null, "hasPregnant": null, "hasInjuredSick": null,
  "injuryNote": null, "vulnerableNote": null,
  "suppliesNeeded": [ {"item":"", "quantity":null, "unit":"", "note":""} ],
  "canMove": "unknown",
  "waterLevel": "unknown",
  "powerComm": ["unknown"],
  "isRareDisease": null,
  "contact": {"name":null, "phone":null, "altContact":null, "canCallBack":null},
  "rareDisease": {
    "diseaseName": null, "diseaseCategory": "unknown", "currentVitals": null,
    "medicationsNeeded": [ {"drugName":"", "dose":null, "remainingAmount":null, "runOutDate":null, "source":""} ],
    "needsDialysis": null, "dialysisNote": null,
    "needsOxygen": null, "oxygenNote": null,
    "coldChainMeds": null, "coldChainNote": null,
    "deviceDependent": [ {"device":"", "powerDependent":null, "consumableNote":""} ],
    "medicalDocs": {"hasDocs": false, "docRefs": []},
    "foundationLinked": null, "allergyContra": null
  },
  "confidence": {"overall": 0.0, "low_fields": []},
  "_uncertain": [],
  "rawText": ""
}

【用户输入】
文字/语音转写：<<<{{USER_TEXT}}>>>
照片：见随附图片（可能为0张或多张）。若有照片，请从中提取：可见水位、被困/受伤迹象、人数线索、门牌店招地标、药盒/病历上的药名与病名、医疗设备。照片看不清的不要猜。

现在只输出该 JSON。
```

**工程加固建议（配合 promptTemplate 使用）：**
- 若模型支持 JSON mode / response_format=json_object 或 structured outputs，务必开启，保证可解析。
- 服务端用 schema 校验（如 Zod）+ 语义校验：研究指出模型可能输出合法 JSON 但 `confidence` 是幻觉（如全填 0.99），需在代码里查分布、对空/枚举越界做纠正与回退。
- 温度调低（0~0.3）以稳定输出。
- 图片压缩到长边 ~1280px 再上传，省 token、省流量（灾区网络差）。
- 语音优先在端上或云函数内转写为文字再喂给多模态模型（多数国产多模态 API 以图文为主）。

## 5. deployment —— 中国大陆免翻墙、静态站不暴露密钥的安全部署

### 5.1 威胁模型
前端是 Vite 静态站（可托管于国内静态托管/对象存储 + CDN）。**任何写进前端的 AI 密钥都会被用户在浏览器里看到**，故 AI 调用绝不能在前端直连。必须经服务端代理，密钥只存服务端环境变量。

### 5.2 推荐架构：BFF 代理（前端 → 云函数代理 → 国产多模态 API）

```
[受灾群众手机浏览器 静态站]
        │  1. 上传照片到 Supabase Storage / 腾讯云 COS，仅拿到临时 URL
        │  2. POST {mediaRefs, userText} 到代理函数（不带任何 AI 密钥）
        ▼
[代理层：Supabase Edge Function  或  腾讯云云函数 SCF]
        │  - AI 密钥存于函数环境变量/密钥管理，前端不可见
        │  - 注入 §4 promptTemplate，调用国产多模态 API
        │  - schema 校验 + 纠错 + 就高定级兜底
        │  - 限流 / 图片大小校验 / 简单风控（防刷）
        ▼
[国产多模态大模型 API（国内可直连、免翻墙）]
        ▼  返回结构化 JSON
[写入 Supabase 数据库（求助表）] → 分派 & 抄送罕见病社区
```

### 5.3 代理层两个选项

**A. Supabase Edge Function（已在技术栈内，改动最小）**
- 项目已用 `@supabase/supabase-js`，天然衔接现有 Supabase 后端。
- 密钥用 `supabase functions secrets set AI_API_KEY=...`，函数内 `Deno.env.get('AI_API_KEY')` 读取，**绝不进浏览器**（官方明确：service/secret key 只能在 Edge Function 用，不能进浏览器）。
- ⚠️ 中国大陆访问风险：Supabase 托管在境外，大陆直连稳定性/合规性存疑（本次搜索未能确证其在华可用性，需实测）。**若面向大陆灾区用户，Edge Function 仅建议做「调度/写库」，把对外的多模态推理调用放到国内云函数**，或整体迁到方案 B。
- 适合：内部/志愿者侧、或已有 Supabase 且大陆访问经实测可接受时。

**B. 腾讯云云函数 SCF / 阿里云函数计算 FC（面向大陆用户首选）**
- 部署在国内可用区，大陆免翻墙、低延迟、便于备案合规。
- AI 密钥存函数环境变量或云密钥管理（KMS/凭据管理），前端只调 HTTP 触发器网关。
- 与国产多模态 API（同厂商云内网更快）天然邻近。
- 数据仍可回写 Supabase（或换国内数据库），前端逻辑不变——只是把「AI 推理代理」这一跳落到国内。
- 适合：主投产路径。

> 折中落地：**前端 → 国内云函数(SCF/FC)做 AI 推理代理 → 国产多模态 API；结构化结果再写回 Supabase（沿用现有互助模块表结构）**。既不暴露密钥、大陆免翻墙，又复用现有 Supabase 数据层。

### 5.4 可选模型（国内可直连、免翻墙；2026 现价参考，需以官网为准）
| 模型 | 厂商 | 备注 / 参考价 |
|---|---|---|
| 通义千问 Qwen-VL / Qwen2.5-VL（qwen-vl-plus/max） | 阿里云百炼 | 视觉理解强、OCR 好，适合读药盒/病历/门牌；qwen-vl-plus 约 ¥0.008/千 token 起。生态成熟、文档全。 |
| 智谱 GLM-4V / GLM-4.6V | 智谱 AI | 原生多模态 + 工具调用，接口响应快；GLM-4V 约 ¥0.05/千 token。 |
| 阶跃星辰 Step-1V / Step 视觉 | 阶跃星辰 | 响应快、支持视频理解；输入约 ¥0.005–0.015/千 token。 |

选型建议：**首选 Qwen-VL 系（OCR/中文场景/生态最稳，利于读药盒病历与方言场景），GLM-4V 作备选/双活兜底**。代理层做成「模型可插拔」（配置切换 + 失败自动切换），避免单点限流导致灾时不可用。

### 5.5 安全与可用性要点
- 密钥仅在代理层环境变量，前端零密钥；开启 HTTPS。
- 代理层加：图片大小/张数上限、单 IP/单设备限流、内容大小限制，防止灾时被刷爆额度。
- 兜底：AI 超时/限流/报错 → 立即回退 §6 引导式表单，前端提示"AI 繁忙，已切换手动填写，同样会被处理"。
- 隐私：病历/病情为敏感个人信息，Storage 做私有桶 + 有效期签名 URL，仅授权救援方可读；日志脱敏。

## 6. fallbackUX —— 未接 AI 时的引导式结构化表单降级

**目标：AI 不可用（未接入 / 限流 / 断网 / 密钥缺失，同 `.env.example` 里互助模块的「降级模式」思路）时，用一套手机端引导式表单覆盖与 §1/§2 完全相同的字段，产出同一份 JSON，无缝进入同一分派流程。**

### 6.1 交互形态：分步问答式（Wizard），一屏一问，大按钮、少打字
1. **第一步 · 你现在最需要什么？**（大图标多选卡片）→ 直接写 `needTypes`。选"被困/涉水"追加水位选择（图示没脚踝/没膝/没腰/齐胸），写 `waterLevel`。
2. **第二步 · 有多危险？**（三档大按钮：马上有生命危险 / 很紧急 / 还能等）→ 结合规则映射 `urgency`（前端也执行"就高"逻辑）。
3. **第三步 · 你在哪里？**：文本框 + "拍一张周围照片/门牌"按钮（拍照仍存 Storage，人工可看）→ `locationText` / `mediaRefs`。可选调用浏览器定位辅助。
4. **第四步 · 都有谁？**：人数步进器 + 老人/小孩/孕妇/伤病/残障 的勾选大按钮 → `peopleCount`/`hasElderly`/`hasChildren`/`hasPregnant`/`hasInjuredSick`/`vulnerableNote`。
5. **第五步 · 能不能动？** 三按钮（能走/要人扶/动不了）→ `canMove`；断电断网多选 → `powerComm`。
6. **第六步 · 需要哪些东西？**：常用物资快捷标签（水/食物/药/衣被/婴儿奶粉尿布/卫生用品…）+ 数量步进 → `suppliesNeeded`。
7. **罕见病/慢病分支**：第一步或单独入口勾选"我/家人有罕见病或需长期吃药" → `isRareDisease=true`，展开 §2 表单：病名（可拍药盒/病历替代打字，OCR 缺席时人工读）、当前状态勾选、所需药物+剩余量+"还能撑几天"、是否需透析/供氧/冷藏药（大按钮 + 备注）。
8. **最后 · 联系方式**（必填电话，缺电话则强提示）→ `contact`；一键提交。

### 6.2 与 AI 版本对齐的关键点
- **同一 JSON 出口**：表单每步直接映射 §1/§2 字段名，提交产出的对象与 AI 版本结构完全一致，`confidence.overall=1`（人工填写视为确定），`_uncertain=[]`，`rawText` 存用户所填自由文本 + "手动填写"标记。
- **就高定级同源**：`urgency` 用同一套规则映射（前端 util 与云函数共享），保证 AI 版与手动版分诊一致。
- **拍照不浪费**：即使不跑 AI，照片仍上传存证，供救援站人工判读——AI 恢复后可对历史照片补跑结构化（异步回填）。
- **可访问性**：大字号、高对比、图标 + 文案双标、支持语音输入到文本框；每步可跳过（跳过=该字段 null），保证「哪怕只填一步也能提交」，最大化触达不便打字/方言用户。
- **零后端也能兜底**：若连云函数都不可达，表单可先写 `localStorage` 暂存并提示"网络恢复后自动提交"，与现有互助模块「本地演示/只读降级」一致。

### 6.3 渐进增强策略
AI 与表单不是二选一，而是**同一表单 + AI 预填**：
- 有 AI：用户拍照 + 说一句话 → AI 预填整张表单 → 用户只需在高亮的 `low_fields` 上快速确认/改正 → 提交。（最省力路径）
- 无 AI：同一张表单，空白起步，用户按引导逐步填 → 提交。
- 两条路径共用一套字段、一套校验、一个提交接口，AI 只是「把手动填写的活儿尽量替用户干掉」，坏了随时退回纯手动，系统永远可用。
